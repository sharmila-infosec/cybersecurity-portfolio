# Incident Response

This folder contains two SOC-level incident response projects completed as part of the Google Cybersecurity Professional Certificate. These projects simulate real Tier 1 and Tier 2 SOC analyst tasks including alert triage and physical security threat analysis.

---

## 📄 Project 1: Alert Ticket — Phishing with Malicious Attachment

### Project Description

In this project, I triaged a medium-severity security alert generated by an IDS/IPS system. The alert indicated that an employee may have received a phishing email containing a malicious attachment. I investigated the alert, verified the file hash using VirusTotal, documented my findings, and made an escalation decision.

### Alert Details

| Field | Details |
|-------|---------|
| Ticket ID | A-2703 |
| Alert Message | SERVER-MAIL Phishing attempt — possible download of malware |
| Severity | Medium |
| Status | Escalated |
| Malicious File | bfsvc.exe |
| File Hash | 54e6ea47eb04634d3e87fd7787e2136ccfbcc80ade34f246a12cf93bab527f6b |
| Sender | Def Communications `<76tguyhh6tgftrt7tg.su>` — suspicious domain |

### Skills Demonstrated

- Alert Triage (Tier 1 & 2)
- Phishing Email Analysis
- IOC Identification
- File Hash Verification (VirusTotal)
- Incident Escalation Decision-Making
- SOC Documentation

### Tools Used

- VirusTotal (hash verification)
- IDS/IPS Alert System
- Incident Ticketing System

### Investigation Summary

- The employee received a phishing email with a password-protected .exe attachment
- The file hash was verified as malicious using VirusTotal
- The sender's domain was suspicious and did not match a legitimate organization
- The alert was escalated to a Level 2 SOC analyst for further investigation and containment

### Key Takeaways

- File hash verification is a fast and reliable way to confirm whether an attachment is malicious
- Suspicious sender domains and password-protected executable files are strong phishing indicators
- Clear escalation documentation helps Tier 2 analysts respond quickly and effectively

## 📄 Project 2: Parking Lot USB Exercise

### Project Description

In this project, I analyzed a USB baiting scenario where an unknown USB drive was found in a hospital parking lot. I examined the drive's contents, applied an attacker mindset to assess potential risks, and recommended organizational controls to prevent USB-based attacks.

### USB Contents

- Family photos and pet pictures (personal files)
- Employee schedules
- New hire letter
- Budget documents

### Skills Demonstrated

- Physical Security Threat Analysis
- Attacker Mindset Thinking
- Social Engineering Awareness
- USB Baiting Attack Analysis
- Risk Mitigation Recommendations
- Security Policy Development

### Attacker Mindset Analysis

An attacker could use the personal and work information on the USB to:
- Launch targeted phishing or social engineering attacks against the employee, their family, or hospital staff
- Use employee schedules and budget documents to gain unauthorized access to hospital systems
- Deploy malware, ransomware, or spyware that auto-executes when the USB is plugged in

### Recommendations

- Train employees never to plug in unknown USB devices
- Disable USB autorun on all organizational endpoints
- Scan removable media in an isolated virtual environment before use
- Enforce endpoint protection software and USB security policies
- Keep personal and work files strictly separate

### Key Takeaways

- USB baiting is a common and effective social engineering attack vector
- A single unknown USB drive can compromise an entire organization's network
- Technical controls combined with employee training are essential for USB threat prevention
